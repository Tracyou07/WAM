# 仓库指南

## 项目结构与模块组织

当前仓库以文档为主。根目录包含 `WM as simulator.pdf`，它应是目前的主要项目产物。后续如果加入源文档、笔记或脚本，请放入清晰命名的目录：

- `docs/`：存放文稿源文件、导出的 PDF 和参考资料。
- `src/`：存放实现代码或仿真程序。
- `tests/`：存放自动化测试。
- `assets/`：存放图片、图表、数据集和其他素材。

避免把临时生成文件放在根目录。文件名应具有描述性；只有在已有文档标题中包含空格时，才保留空格。

## 构建、测试与本地开发命令

当前没有构建系统。根目录中未发现 `package.json`、`Makefile`、Python 项目文件或测试运行器配置。

如果后续加入代码，请在本节补充对应命令，例如：

- `npm test`：运行 JavaScript 或 TypeScript 测试。
- `python -m pytest`：运行 Python 测试。
- `make build`：生成构建产物。

仅修改文档时，请确认导出的 PDF 能正常打开，并检查图表、引用和页面布局是否正确。

## 编码风格与命名约定

后续引入任何语言或工具链时，应遵循对应生态的常用约定。模块应保持聚焦，命名应说明其领域行为。目录名建议使用小写形式，例如 `src/`、`tests/` 和 `assets/`。

Markdown 文档应使用清晰标题、带语言标记的代码块和简短段落。脚本中不要硬编码配置值；必要的环境变量应写入文档。

## 测试指南

当前没有自动化测试。添加代码时，请将测试放在 `tests/` 下，并使用能对应实现文件的命名方式，例如 `test_simulator.py` 或 `simulator.test.ts`。

文档变更应通过打开最终 PDF 进行检查，重点确认格式、图表位置、参考文献和重新生成的产物。

## 提交与拉取请求规范

当前目录不是 Git 仓库，因此没有可参考的本地提交历史。如果后续初始化版本控制，请使用简短的祈使句提交信息，例如 `Add simulator notes` 或 `Update document assets`。

拉取请求应包含简洁摘要、变更文件、已执行的验证步骤；涉及版式变化时，应附截图或导出的 PDF。若有关联 issue 或设计说明，也应一并链接。

## 安全与配置建议

不要提交凭据、私有数据集或仅适用于本机的路径。大型生成文件只有在确属项目产物时才应纳入仓库，并尽可能说明其重新生成方式。
